    Your current validator screen
    
              
    If the score is high but the status says Needs QA, that is correct. The build may be strong, but warnings still need review before the build can be called complete.
    
              
    
                
    Score 95Strong structure. Not automatic completion.
    
                
    Needs QAReview warnings before upload or launch.
    
                
    Complete allowed: NoThe no false complete rule is active.
    
              
    
            
    

            
    
              
    What passed
    
              
      
                
    • Manifest readiness passed. The app metadata looks correct.
      • 
                
    • Offline support passed. Install, fetch, and cache logic were found.
      • 
                
    • Assets passed. Images and references were detected.
      • 
                
    • Promised files passed. No obvious missing promised file evidence was found.
      • 
              
    
            
    

            
    
              
    What needs review
    
              
      
                
    • Nested folder warning. The app may be inside a folder like witness-chef/index.html. GitHub Pages may need the folder root adjusted.
      • 
                
    • Placeholder links. href="#" links may look clickable but may not go anywhere.
      • 
                
    • Duplicate ID. A repeated ID such as bookingOccasion can break forms or JavaScript.
      • 
                
    • Missing CSP. A Content Security Policy should be added for stronger static-site security.
      • 
              
    
            
    

            
    
              
    What to do next
    
              
      
                
    1. Go to Fix Studio or Repair Command Center.
      2. 
                
    2. Look for cards about nested folder, duplicate ID, placeholder links, and CSP.
      3. 
                
    3. Use Prepare Fix first. This lets you preview what will happen.
      4. 
                
    4. Use Apply Safe Patch only for safe or safe-with-review items.
      5. 
                
    5. Use Create Repair Pack to download repaired files and reports.
      6. 
                
    6. Do not mark the build complete until the hosted site is tested.
      7. 
              
    
            
    

            
    
              
    Safe repair order
    
              
      
                
    1. Fix nested folder or GitHub Pages path first.
      2. 
                
    2. Fix duplicate IDs second.
      3. 
                
    3. Review placeholder links third.
      4. 
                
    4. Add CSP fourth.
      5. 
                
    5. Export a Repair Pack.
      6. 
                
    6. Upload the repaired files.
      7. 
                
    7. Run the validator again.
      8. 
                
    8. Mark stable only after live QA passes.
      9. 
              
    
            
    

            
    
              
    Plain meaning of each warning
    
              
    
                
    Nested folderThe site may work only if uploaded to the right folder path.
    
                
    href="#" linksSome buttons or links may not have real actions yet.
    
                
    Duplicate IDTwo fields may share one name, which can confuse forms and scripts.
    
                
    No CSPThe site needs a stronger browser security rule.
    
              
    
            
    

            
    
              
    When can a build be marked stable?
    
              
    Only after these checks pass:
    
              
      
                
    • The hosted page opens without a 404.
      • 
                
    • The splash page appears first.
      • 
                
    • The main buttons work.
      • 
                
    • The Home Screen icon is correct.
      • 
                
    • The validator no longer shows critical errors.
      • 
                
    • Any patched issue has been tested live.

    Repo File Worker — Phase 2 Preview
    GitHub Action Spec
    How the secure executor will work in Phase 2. No browser token. No direct write. No merge.
    
          
    
            
    Architecture: What Each Layer Does
    
            
    
              
    Load Tasks (this app)
    Creates structured repo-file-job.json. Classifies risk. Drafts PR. No tokens. No direct writes. Output is a JSON file the user exports.
    
              
    User
    Reviews the repo-file-job.json. Approves or rejects. If approved, uploads JSON to trigger the GitHub Action. All merges require explicit user approval. User is the release gatekeeper.
    
              
    GitHub Actions (Phase 2 — not yet built)
    Reads the approved repo-file-job.json securely (no browser exposure). Creates a new feature branch from latest main. Applies the approved file changes. Opens a PR. Does NOT merge. Does NOT deploy.
    
              
    GitHub Repo
    Source of truth for all files. PR target. Pages host. Protected main branch. All changes land via PR — never direct push.
    
            
    
          
    
          
    
            
    Phase 2 GitHub Action: Planned Behaviour
    
            
    
              
    1. Trigger: dispatch event with repo-file-job.json payload
    
              
    2. Validate: check riskLevel, serviceWorkerImpact, userApprovalRequired fields
    
              
    3. Reject HIGH-risk jobs unless explicit override flag is present in the JSON
    
              
    4. git fetch origin main
    
              
    5. git checkout -b <branchNameSuggestion>
    
              
    6. Apply file changes as defined in the job (patch or full replacement)
    
              
    7. git commit -m "<prTitleSuggestion>"
    
              
    8. git push -u origin <branch>
    
              
    9. Open PR with prTitleSuggestion and prBodyDraft — NOT auto-merged
    
              
    10. Report PR URL back to user. Wait for user to review and merge.
    
            
    
          
    
          
    
            
    Hard Rules — Never Implemented in Phase 2
    
            
    
              
    — No GitHub token stored in the browser
    
              
    — No Cloudflare token anywhere
    
              
    — No direct push to main
    
              
    — No auto-merge without explicit user approval
    
              
    — No file deletion without explicit per-file approval in the job
    
              
    — No branch deletion
    
              
    — No production deployment trigger
    
              
    — No service worker / cache changes without HIGH-risk approval flag
    
              
    — No AI provider routing changes without HIGH-risk approval flag

    Repo File Worker — Phase 2 Preview
    Phase 2 Executor Readiness Checklist
    Confirm all conditions before enabling the GitHub Actions secure executor.
    
          
    
          
    
            
    Phase 2 Readiness
    
            
    
            
    
          
    
          
    
            
    Phase 2 Architecture
    
            
    
              
    Flow: LoadTasks (browser) saves job JSON to localStorage → user exports JSON → commits as repo-file-job.json to a feature branch → GitHub Actions reads the job, opens a PR with the specified changes → user reviews + merges.
    
              
    Hard rules for Phase 2 executor:
    
              
    
                
    — No GitHub token in the browser, ever
    
                
    — No push to main
    
                
    — No merge without user approval
    
                
    — No delete operations
    
                
    — No branch cleanup
    
                
    — No deployment trigger
    
                
    — No SW/cache changes without HIGH-risk flag
    
                
    — No AI provider routing changes without HIGH-risk flag
    
                
    — Job JSON is read-only input; executor cannot modify jobs
    
                
    — Executor creates a PR and stops; user is the release gatekeeper
